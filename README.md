# saudi-laws-md

نصوص الأنظمة واللوائح السعودية بصيغة Markdown نظيفة، منظمة حسب المجال، للبحث والاستخدام في أدوات المطورين والباحثين.

الانظمة: https://github.com/abosalehg-ui/saudi-laws-md/tree/main/laws

## ⚠️ تنويه مهم

هذا المستودع **غير رسمي** وغير تابع لأي جهة حكومية، وهو تجميع شخصي لنصوص الأنظمة السعودية.

**لا يُعتد بهذا المستودع كمصدر رسمي.** للتحقق من دقة أي نص أو معرفة آخر التعديلات، يُرجى الرجوع إلى:
- [جريدة أم القرى الرسمية](https://uqn.gov.sa)
- [بوابة الأنظمة السعودية (هيئة الخبراء بمجلس الوزراء)](https://laws.boe.gov.sa)

## المحتوى

يضم `laws/` نحو 2840 ملف Markdown مصنّفة في مجلدات حسب المجال (`أنظمة العمل والرعاية الاجتماعية`، `أنظمة الصحة`، `أنظمة الأمن الداخلي...`) أو نوع الوثيقة (`نظام`، `لائحة`، `مرسوم ملكي`، `أمر ملكي`، `قرار`، `اتفاقية`، `معايير`). الملفات التي لم يُحدَّد تصنيف دقيق لها بعد تُجمَّع في `laws/غير-مصنف/`.

عند وجود وثيقة من كلا المصدرين (qanoonsa وnezams) تُبقى نسخة واحدة (يُفضَّل مجلد المجال وبيانات nezams الأغنى) ويُحفظ رابط النسخة الأخرى في حقل `also_available_from`.

المجلدات المكررة بفروق إملائية/همزة/ترتيب كلمات جرى دمجها في مجلداتها المعتمدة (انظر [CHANGELOG](CHANGELOG.md)). يبقى تحدٍّ بنيوي أعمق: أغلب ملفات مجلدات النوع (`قرار`، `مرسوم ملكي`، `أمر ملكي`، `اتفاقية`، `معايير`) لا تحمل تصنيف مجال في بياناتها الوصفية، وكثير منها (قرارات إجرائية/أوامر تعيين وإعفاء) بلا مجال موضوعي طبيعي، فتبقى منظّمة حسب النوع؛ `scripts/reclassify.py` يعالج ما يمكن تصنيفه تدريجيًا.

كل نظام مخزَّن كملف Markdown مستقل يحتوي:
- بيانات الإصدار (المرسوم/القرار، التاريخ الهجري والميلادي، رقم عدد أم القرى) في الـ front matter
- النص الكامل مقسّمًا حسب الأبواب والمواد
- سجل التعديلات التاريخية لكل مادة (عند توفره)

## صيغة الملف

كل نظام يتبع القالب التالي:

```markdown
---
title: اسم النظام
doc_type: نظام
issued_by: ...
approval_date_hijri: ...
publish_date: ...
gazette_ref: ...
status: نافذ / معدّل
category: ...
attachments: [...]
---

# اسم النظام

## المادة الأولى
...

> **تعديلات المادة:**
> - ...
```

قسم "تعديلات المادة" يظهر فقط عند توفر سجل تعديلات تاريخي لتلك المادة.

## الأداة (scripts/)

أداة Python تسحب صفحات الأنظمة من مصدرين — [قانون (qanoonsa.com)](https://qanoonsa.com) و[نظام (nezams.com)](https://nezams.com) — وتحوّلها إلى القالب الموحد أعلاه.

### التثبيت

```bash
pip install -e ".[dev]"
```

### الاستخدام

```bash
# سحب من رابط (المصدر يُكتشف تلقائيًا من النطاق)
python -m scripts.main "https://nezams.com/نظام-العمل/"

# عدة روابط من ملف (رابط في كل سطر)
python -m scripts.main --from-file urls.txt

# وضع بدون شبكة: صفحة HTML محفوظة من المتصفح
python -m scripts.main --html page.html --source nezams --url "https://nezams.com/..."

# خيارات: --out laws  --category عمل  --delay 1.5
```

الملفات تُكتب إلى `laws/<التصنيف>/<اسم النظام>.md` (والتصنيف غير المتوفر يذهب إلى `laws/غير-مصنف/`). الصفحات التي لا تطابق البنية المتوقعة تُسجَّل في `logs/failed.txt` ولا توقف التشغيل، وتطبع الأداة تحذيرًا عند وجود خلل في تسلسل أرقام المواد.

### الاكتشاف والاستيراد بالجملة

**أمر واحد** يكتشف الروابط من خرائط الموقعين ثم يحوّلها على دفعات مؤدَّبة قابلة للاستئناف، ويكتب تقريرًا:

```bash
python -m scripts.main --discover qanoonsa,nezams --resume --limit 50 --delay 2 --report
#   كرّر الأمر نفسه في جلسات لاحقة؛ يستأنف من حيث توقّف تلقائيًا حتى تكتمل كل الروابط.
```

- `--discover qanoonsa,nezams` يكتشف كل روابط الوثائق مباشرةً (بدل خطوة منفصلة وملف وسيط). لإدراج صفحات تعديلات المواد المفردة في nezams: أضِف `--include-updates`.
- `--resume` يسجّل كل رابط ناجح في `logs/done.txt` ويتخطّاه في التشغيلات التالية، فيمكن تقسيم آلاف الروابط على جلسات متعددة دون إعادة عمل.
- `--limit N` يوقف الدفعة بعد `N` رابطًا جديدًا (يناسب حدود السياق في الجلسة الواحدة).
- `--delay` يفرض تأخيرًا بين الطلبات احترامًا للموقعين (كلاهما يسمح بالزحف حسب `robots.txt`).
- `--report` يكتب `logs/summary.md`: عدد النجاح/الفشل/التخطّي، توزيع الأنواع، كل التحذيرات مجمّعة، وقائمة الفشل بأسبابها — بدل تتبّع stderr و`logs/failed.txt` يدويًا.

لا يزال بالإمكان الاكتشاف وحده (لمعاينة الروابط): `python -m scripts.discover --out urls.txt`.

التصنيف يُشتق تلقائيًا: تصنيف nezams الخام يُبسَّط إلى مجال مقروء، وصفحات qanoonsa بلا تصنيف تُنظَّم حسب `doc_type`، فيقلّ ما يتكدّس في `laws/غير-مصنف/`.

### التحقق من تحديثات لاحقة (بلا إعادة سحب كاملة)

لروابط سبق استيرادها، `--check-updates` يرسل `ETag`/`Last-Modified` المحفوظين من آخر جلب (حقول `etag`/`last_modified` في الملف نفسه)، ويتخطّى إعادة التحليل والكتابة إن ردّ الموقع بأن المحتوى لم يتغيّر (304):

```bash
python -m scripts.main --discover qanoonsa,nezams --check-updates --report
```

يوفّر هذا نطاقًا تردديًا عند إعادة زيارة آلاف الروابط دوريًا للتحقق من تعديلات لاحقة على الأنظمة، ويتجنّب تغييرات زائفة في git (تحديث `retrieved_at` لملفات لم يتغيّر محتواها فعليًا). لروابط جديدة لا يوجد لها ملف سابق، يُسحَب المحتوى كاملًا كالمعتاد بلا شرط.

### صيانة وفحص المخرجات الحالية

سكربتات صيانة تعمل على `laws/` مباشرة دون إعادة سحب من الشبكة:

```bash
python -m scripts.reclassify laws --dry-run          # يعيد حساب doc_type/category وينقل الملفات عند الاختلاف
python -m scripts.audit_duplicates laws --dry-run     # يربط الوثائق مزدوجة العنوان عبر حقل also_available_from
python -m scripts.canonicalize_urls laws --dry-run    # يطبّع صيغة الروابط (فك ترميز) في source_url/also_available_from
python -m scripts.lint_corpus laws                    # فحص صحة المُدوَّنة: أخطاء صلبة (رمز خروج ≠0) وتحذيرات
python -m scripts.build_index laws                    # يبني index.json (فهرس آلي بالبيانات الوصفية)
```

أزل `--dry-run` للتنفيذ الفعلي. سكربتات النقل/الربط لا تحذف محتوى ولا تدمج ملفات تلقائيًا عند وجود تعارض؛ التعارضات تُطبع للمراجعة اليدوية.

- **`lint_corpus`** يفصل الأخطاء الصلبة (عنوان معطوب، ضجيج واجهة، فقرات مكرّرة، حقل إلزامي ناقص — تُفشل CI) عن التحذيرات (تصنيف ناقص/غير مطابق للمجلد، خلل تسلسل — تُعرض بـ `--warnings`).
- **`build_index`** ينتج [`index.json`](index.json): مدخل لكل وثيقة (العنوان، النوع، التصنيف، الحالة، المصدر، الرابط، المسار) لأدوات البحث وRAG دون مسح آلاف الملفات؛ `--check` يتحقق أنه محدَّث (يعمل في CI).
- **`canonicalize_urls`** يوحّد صيغة الروابط بعد اختلافها تاريخيًا بين مُرمَّز/غير مُرمَّز، فتتطابق آليات الهوية (الاستئناف، الجلب الشرطي، كشف التكرار).

### احترام robots.txt

تحترم الأداة `robots.txt` لكل مضيف افتراضيًا (تُجلَب مرة وتُخزَّن)؛ لتعطيله أضِف `--ignore-robots`.

### كشف نوع الوثيقة والتحقق

- **النوع** يُكتشف تلقائيًا من العنوان ويُكتب في حقل `doc_type` بالـ front matter: `نظام`، `لائحة`، `مرسوم ملكي`، `أمر ملكي`، `قرار`، `اتفاقية`، `معايير`، `تعديل`، أو `أخرى`.
- **الوثائق غير المقسّمة لمواد** (الأدلة والمعايير والجداول) لم تعد تُعدّ فشلًا؛ يُحفظ متنها كـ Markdown (مع تحويل الجداول) في حقل نصّي.
- **التحقق** (`validate_document`) يرصد — دون إيقاف التشغيل — خللَ تسلسل المواد، وبقايا ضجيج الواجهة، والمواد الفارغة، والوثائق الخالية من المحتوى.

### الاختبارات والفحص

```bash
python -m pytest
ruff check scripts tests
```

CI (GitHub Actions):
- `tests.yml` على كل push/PR: `pytest` (على Python 3.10/3.11/3.12)، `ruff`، فحص `lint_corpus`، والتأكد أن `index.json` محدَّث.
- `update-corpus.yml` (شهري + يدوي): جلب شرطي من المصدرين ثم فتح PR مسودة بالتغييرات للمراجعة.
- `smoke.yml` (أسبوعي): فحص دخاني ضد المواقع الحيّة يكشف تغيّر بنية HTML مبكرًا.

## التعديلات على الأنظمة

بعض المواد تصدر بنسخ معدّلة لاحقًا؛ عند توفر سجل تعديلات تاريخي لمادة ما، يُحفظ ضمن نفس ملف النظام في حقل `amendment_history` (يظهر كقسم "تعديلات المادة" أسفل كل مادة).

لا يوجد مجلد أرشيف منفصل للنسخ الكاملة السابقة، لكن التحديث الدوري (`update-corpus.yml`) يعتمد الجلب الشرطي فلا يعيد كتابة الملفات إلا عند تغيّر محتواها فعليًا؛ وبذلك يصبح **تاريخ git نفسه** أرشيفًا زمنيًا لكل تعديل لاحق على أي نظام (يمكن استرجاع أي نسخة سابقة عبر `git log`/`git show` على مسار الملف).

## المساهمة

المساهمات مرحّب بها، خصوصًا في:

- إضافة أنظمة جديدة
- تحسين التصنيف (دمج المجلدات المتقاربة الاسم، تقليص `laws/غير-مصنف/`)

يُرجى فتح Issue قبل أي تعديل جوهري على البنية.
